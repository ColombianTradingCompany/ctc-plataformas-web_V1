## What this is

Colombian Trading Company (CTC) is a green-coffee exporter. This repo is a single Next.js app serving **four surfaces** on one deployment, routed by subdomain in production and by path in dev:

| Platform | Audience | Route | Subdomain (prod) |
|---|---|---|---|
| **CTC Home** | Public marketing | `/` | root domain |
| **Kaffetal Regal** | Coffee producers | `/kaffetal-regal` | `kaffetal-regal.*` |
| **Cherry Picked** | European roasters/buyers | `/cherry-picked` | `cherry-picked.*` |
| **BCP** (Business Control Panel) | CTC's own staff (2 founders) | `/bcp` | not on a public subdomain |

The subdomainâ†’path rewrite lives in `src/proxy.ts` (see Gotchas below for why it's not called `middleware.ts`).

**The core business flow this app encodes**: a producer registers a finca + lot and fills out a Ficha TÃ©cnica (Kaffetal Regal) â†’ CTC staff schedule an Arena cupping session and grade it (BCP) â†’ a graded lot gets a purchase contract with a 3-month release staircase (BCP; **signing requires the producer to be a Kaffetal Club member** â€” BCP emits a member code in /bcp/club, the producer redeems it in "Mis contratos") â†’ once at least one month has released, BCP publishes it to the Cherry Picked catalog (also member-gated) â†’ a European buyer reserves and checks out (Cherry Picked).

## Supabase client factories (`src/lib/supabase/server.ts`)

Three, each for a different trust level â€” know which one a piece of code should use before touching auth/data code:

1. `createSessionClient()` â€” cookie-bound, respects RLS as the signed-in user. Default choice for anything a logged-in user does to their own data.
2. `createEphemeralClient()` â€” no session persistence, used only for the BCP login's password-verify-without-cookie step.
3. `createServiceRoleClient()` â€” bypasses RLS entirely. **Server-only, never import from a client component.** Reserved for audited admin mutations (BCP approvals/publishing) and system-computed fields a normal user JWT isn't allowed to touch (e.g. the AI-advice cache columns on `lots`, written even when the lot is past the stage a producer's own JWT can modify).

### The "ironclad guard" trigger model

Every table a normal user JWT can write to has a guard: RLS restricts *whose* rows are visible, and a `BEFORE UPDATE` trigger additionally restricts *which columns/transitions* are legal, keyed off `auth.uid() is not null` (service-role calls have a null uid, so BCP/system writes bypass the guard on purpose). Current guards, all confirmed intact and matching their trigger source during the 2026-07-10 audit:

| Table | Guard function | Producers/buyers cannot... |
|---|---|---|
| `profiles` | `guard_profiles_protected_columns` | change their own `role` or `email` |
| `buyer_profiles` | `guard_buyer_protected_columns` | self-assign `lifetime_points`/`membership_tier` (the points trigger sets `app.syncing_points` to bypass this for its own writes) |
| `lots` | `guard_lots_producer_columns` | touch a lot once its `stage` is past `ficha_completa`, change `stage` outside `{borrador, ficha_completa}`, or ever set `grade` |
| `lots` (2nd guard) | `guard_lot_protected_columns` | change `sample_2kg_confirmed_at`, `source`, `producer_id`, or make any stage transition other than `borrador â†’ ficha_completa` (added 2026-07-13; overlaps the older guard on purpose â€” both run, effective rule is the intersection) |
| `fincas` | `guard_finca_protected_columns` | change `status` (no self-approval), `eudr_cert_shared` (no self-release of the certification), `producer_id`, or any of BCP's evaluation columns (`eudr_evidence_*`, `eudr_legal_areas`, `eudr_sustainability_*`); **and once `status = 'approved'`, their own EUDR declarations (deforestation/legal/tenure/planting/system/lat/lng/polygon/hectares/producer_answers) are locked** â€” changes go through "Solicitar revisiÃ³n de datos" (added 2026-07-13; keyed off `auth.role()` not in `{service_role, postgres}`) |
| `producer_profiles` | `guard_producer_protected_columns` | self-grant `club_member_since` (Kaffetal Club membership) â€” guarded on **INSERT too**, because `producer_profiles_insert_own` + the InfoModal upsert path means a producer can create their own row (added 2026-07-14) |

If you add a new producer/buyer-writable column to one of these tables, check whether it needs carving out of (or into) its guard function â€” the AI-advice cache columns deliberately go around this guard via the service-role client rather than punching a hole in it.

## Gotchas (learned the hard way this project â€” don't re-learn these)

1. **Two Supabase projects exist on this org; only one is real.** `sjznkzvefqfcysczllli` ("ctc-platform") is `ACTIVE_HEALTHY` and is what `.env.local` actually points at. `dweizejazqeqpiguecxw` ("ColombianTradingCompany's Project") is a long-abandoned early-exploration project, paused >90 days, **unrecoverable**, and not used anywhere. If any Supabase MCP tool call times out with "Connection terminated," check you didn't default to the wrong `project_id` â€” this happened once and wasted several tool calls before the mismatch was spotted via `list_projects`.
2. **`src/proxy.ts` is this Next.js version's `middleware.ts`.** Not a typo, not custom infra â€” see `AGENTS.md`'s warning that this Next.js build has real breaking API changes from training-data Next.js.
3. **React `useEffect` + `react-hooks/set-state-in-effect`**: this lint rule flags ANY direct synchronous `setState()` at the top level of an effect body, even before an async chain. Fix pattern used throughout: seed the "loading"/initial state from props at `useState()` time so the effect never needs a synchronous reset call, and only ever chain `.then(handler)` off a promise in the effect body.
4. **`onAuthStateChange`'s `SIGNED_IN` event fires not just on a fresh login but also when it resolves an already-valid session from cookies** (tab refocus, background token refresh). A handler that unconditionally forces the UI back to a "logged in" landing view on `SIGNED_IN` will silently kick a user out of whatever sub-view they were in (this actually happened in Kaffetal Regal's Ficha view â€” fixed by only forcing the view when coming from the logged-out state, never unconditionally).
5. **Prompt caching (`cache_control`) has a minimum cacheable-prefix token count** (~1024â€“2048 tokens depending on model tier) â€” a short system prompt below that silently never caches (no error, just always misses). Don't assume `cache_control` alone solves a token-cost problem; check the actual prompt size first.
6. **Auto-mode classifier blocks, by design, in this environment**: direct `git push` to `main` (needs the user to push themselves, or a feature branch + PR), Supabase `restore_project` on a shared project, and applying DB migrations that go beyond what was explicitly asked (even safe, additive hardening). When blocked, explain what you intended and let the user decide â€” don't work around it.
7. **Ficha preview HTML is built via raw string templates + `dangerouslySetInnerHTML`** (`fichaPreviewHtml.ts` â†’ `FichaPreview.tsx`) but is properly escaped via a dedicated `esc()` helper on every interpolated field â€” confirmed safe during the 2026-07-10 audit. If this rendering approach is ever reused elsewhere (e.g. surfacing Ficha text to BCP or buyers), re-verify escaping travels with it.
8. **Animated media must NOT go through `next/image`** (2026-07-14) â€” it rasterizes the animation to its first frame, silently. The two site loops (CTC Home's hero `hero-guacamayo-finca.webp`, Kaffetal Regal's `hero-zoom-cereza.webp`) use a plain `<img>` with an `eslint-disable-next-line @next/next/no-img-element` and a comment saying why. Both were **converted GIF â†’ animated WebP with Pillow** (7.3 MB â†’ 2.2 MB and 5.7 MB â†’ 1.6 MB): a multi-MB GIF on an above-the-fold element is an LCP problem, `optimize=True` on the GIF itself makes it *bigger*, and there's no ffmpeg on this machine (so no MP4/WebM route). Also: **the source loops are only 400Ã—225**, so any full-bleed use upscales ~5Ã— â€” they're deliberately blurred slightly and given short/wide frames rather than tall ones (this is exactly why KR's loop is a slim `TopBanner` and not a hero backdrop). Check new generator output for watermarks before shipping â€” the cherry loop had a "Veo" mark that had to be cropped out.
9. **Text over these loops needs a measured scrim, not a guessed one.** Both heroes compute contrast against the *worst-case* composite (brightest/darkest frame under the scrim): CTC Home is light-on-dark â€” the scrim was **lightened 2026-07-14** ("too dark" feedback) and re-measured per element band against every frame of the loop (Pillow script compositing the actual CSS gradients): h1 5.5:1, lead 5.8:1, facts 5.2:1, hero gold 4.6:1 worst-case; the lead/facts hexes were brightened and the hero got a local gold (`#F7D287`, `--accent-soft` no longer clears 4.5:1 there) to buy that headroom. Kaffetal Regal stays light-themed with a near-opaque paper wash (h1 12.7:1, and the `--muted` lead needed the wash pushed to .965 to clear AA at 5.1:1). If you retint either scrim, re-measure the small mono text and the gold â€” they're the ones that fail first.
