Clientes — las 4 superficies que la gente toca
🏠

CTC Home

Público

Marketing corporativo, terminado, sin backend. El único JS real abre un modal de contacto que arma un mailto: — nunca toca la base de datos.

El pipeline de leads dentro de BCP (/bcp/leads) sigue siendo un placeholder "Próximamente" — por eso CTC Home no tiene ningún cable de escritura hacia el resto del sistema todavía.

🌱

Kaffetal Regal

Productor

Portal del caficultor: login real (email+contraseña o Google), CRUD de fincas/lotes, y la Ficha Técnica de 8 paneles con cálculos en vivo. La superficie más activa del proyecto ahora mismo.

Perfil ("Información general") ampliado 2026-07-10: cédula cafetera, celular+WhatsApp, foto y video del productor, país/departamento base, y un código de proveedor autogenerado exportable como QR — ver el nodo Supabase Storage.

☕

Cherry Picked

Comprador

Vitrina para tostadoras europeas: catálogo alimentado solo por publicaciones de BCP, carrito, checkout atómico, niveles de membresía, sample packs.

La subasta Tyrian (TyrianSection.tsx) sigue siendo 100% demo estática — deliberadamente aplazada, aún no empezada.

🛡️

BCP · Business Control Panel

Staff CTC

La única superficie que usa el propio equipo de CTC: login 2FA, aprobación de fincas, Arena, contratos, publicación al catálogo. No enlazada desde ningún nav público; robots.ts la bloquea.

Frontend y entrega — cómo el código llega al navegador
🧩

Código Next.js

Código

Un solo repo, cuatro caras. App Router + TypeScript + React 19, Turbopack. CSS Modules a mano en las 3 superficies públicas; shadcn/ui + Tailwind solo dentro de BCP.

🔀

proxy.ts

Enrutador

El middleware.ts de versiones anteriores de Next.js se llama proxy.ts aquí. Lee el hostname, mapea subdominio → ruta interna y reescribe.

🌍

Vercel

Hosting + CI/CD

Build (Turbopack) + CDN + deploy automático en cada git push a main. Un solo proyecto sirve el dominio raíz y ambos subdominios.

Identidad y acceso a datos — cómo cada plataforma habla con Supabase
🔑

Supabase Auth

Identidad

Dos esquemas conviven: productores/compradores usan email+contraseña o Google OAuth normales; BCP envuelve eso con su propio password+OTP de 2 factores.

🔌

Cliente Supabase (navegador)

Acceso directo

Kaffetal Regal y Cherry Picked llaman a supabase-js directo desde el navegador — no existe una API intermedia para ellos. La seguridad vive entera en RLS + guardas.

📜

Server Actions (BCP)

Contrato admin

BCP nunca toca Supabase desde el navegador. Cada mutación es una Server Action que corre requireAdmin() por su cuenta, más 2 route handlers dedicados (2FA, asesor IA).

Lógica de negocio — dónde viven las reglas reales
🧱

RLS + guardas "ironclad"

Reglas

Dos capas: RLS decide de quién son las filas visibles; un trigger BEFORE UPDATE decide además qué columnas/transiciones son legales.

Las llamadas con service-role tienen auth.uid() nulo — por diseño, la guarda las deja pasar sin más.

⚙️

Funciones SECURITY DEFINER1 de 5 intencional

Lógica atómica

place_order() es la única invocada a propósito (desde el navegador del comprador). Las otras 4 son triggers automáticos que Postgres expone sin querer vía RPC.

🤝

Servicios externos

Rentado

Resend entrega el código OTP de BCP. La API de Anthropic (fetch crudo, sin SDK) resuelve el asesor "¿Y ahora qué?" de la Ficha Técnica, con caché por lote.

Datos — dónde se recuerda todo
🗄️

Postgres 23 tablas

Fuente de verdad

Agrupadas por dominio: Identidad, Kaffetal (fincas/lots/ficha), Arena/Contratos (service-role-only), Cherry Picked (listings/orders/puntos), Ops (OTP/audit_log).

🔎

Vistas públicas

Lectura filtrada

public_lot_catalog y public_transparency_pricing: existen para no tener que ensanchar el RLS de lots/fincas.

📦

Supabase Storagebucket kaffetal-media

Archivos

media_assets + el bucket privado kaffetal-media (activado 2026-07-10, límite 100 MB/archivo): foto y video del productor, video de cada finca, video del café por lote, y adjuntos de soporte por certificado en A3/A4 (≤5 MB).

Todo path empieza por {producer_id}/... — las 4 políticas RLS de storage.objects solo comparan ese primer segmento contra auth.uid(), así que un upload nuevo no necesita una política nueva mientras respete esa forma.
