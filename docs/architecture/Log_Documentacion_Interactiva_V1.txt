## [2026-07-10] Kaffetal Regal producer-profile + Ficha Técnica rework (commit 520180b)
- Producer profile ("Información general") expanded: cédula cafetera, celular + WhatsApp
  toggle, foto de perfil, video del productor y su equipo, país, departamento base, and
  an auto-generated supplier code exportable as a QR (client-side, qrcode package).
- New private Supabase Storage bucket "kaffetal-media" backs every upload (avatar,
  producer video, finca video, lot video, certificate support attachments), via
  src/lib/kaffetalMedia.ts. Path convention: {producer_id}/... everywhere.
- Ficha A1: Proveedor/NIT/Productor now read-only, sourced from the profile. Especie
  moved from A1 to B1.
- Ficha A2: selecting a real finca now makes país/departamento/municipio/masl/geo
  read-only and permanently synced to that finca's record (previously only filled
  blanks once).
- Ficha A3/A4: every certificate got an info tooltip + a support-document upload
  (stored in a new cert_attachments map on FichaFormData, not new columns).
- Ficha footer: CSV buttons removed, replaced by Guardar (progress-only) and
  Completar y Enviar (gated on A1+A2+B1; only this can advance stage past
  ficha_completa, via a new `finalize` flag).
- NextStepWidget ("¿Y ahora qué?") moved below the footer, no longer auto-fires.
- B4 restructured to just "Video del Café" (wired to the previously-unused
  lots.video_asset_id); old B4 notes/Q-Grader moved into B2; producer/finca videos
  relocated to where they belong.
- Sample-shipping address corrected across CTC Home and Kaffetal Regal.

  Doc changes this should produce at the next wrap: update n-kaffetal node
  (note about the profile expansion + storage), rewrite n-storage node (was
  "0 filas, gap conocido" -> now describes the real kaffetal-media bucket and
  its RLS path convention), add n-supaclient -> n-storage wire (the
  direct-upload path is now real, not speculative), update the "kaffetal" and
  "storage" DICT entries to match, add a new scenario "perfil" (Productor
  completa su perfil y sube archivos).

## [2026-07-10] Follow-up fixes, round 1 (commit 8ce14a0)
- Fixed a real production crash: old lots with a saved datasheet from before
  cert_attachments existed threw on A3/A4 mount (`undefined[key]`). Fixed by
  merging onto EMPTY_FICHA instead of an either/or fallback -- this matters for
  *any* future field added to FichaFormData, not just this one.
- "Completar y Enviar" was a real HTML disabled button (silently did nothing
  when incomplete) -- switched to aria-disabled + dimmed styling, still
  clickable, now shows a toast naming what's missing.

  Doc changes: none directly (this was a bugfix to already-documented
  behavior, not a new flow) -- captured here so the wrap history is complete;
  worth a one-line mention in the n-kaffetal or n-guards node text in a
  future wrap if it comes up again.

## [2026-07-10] EUDR due-diligence module: Finca + Lote + BCP (commit pending)
- New `src/lib/eudr.ts`: pure, dependency-free `fincaEudrStatus()`/`lotEudrStatus()`
  ported from the reference "EUDR Ready · Café" prototype's fincaEstado()/loteEstado(),
  reinterpreted onto this system's real Finca/Lot records. Shared verbatim between
  the Kaffetal Regal client UI and BCP's server-rendered review pages.
- Finca-level (migration eudr_finca_fields): geolocation (lat/lng), fecha de
  establecimiento, sistema productivo, declaración de no deforestación + de
  producción legal, evidencia, áreas de legislación, tenencia, sostenibilidad.
  New "EUDR · Debida diligencia" section in FincaModal.tsx with a live status
  badge. Also fixed a pre-existing gap found while building this: the old
  "Geolocalización" field was never actually saved (dbFincaToFinca hardcoded
  geo:"" and saveFinca never sent it), so BCP's requires_eudr_polygon gate was
  silently inert -- now real lat/lng round-trips both ways.
- Lote-level (migration eudr_lot_fields): cadena de custodia, evaluación de
  riesgo (Art. 10(2)), mitigación -- new Ficha pane A5 ("EUDR / Debida
  Diligencia"), wired into FichaNav/FichaView alongside a live status badge.
  `additional_estates: string[]` (freetext finca names, used by Regional/
  Multi-Origin blends) renamed to `additional_estate_ids: string[]` (real
  finca ids) -- without that fix a blend's other fincas had no real EUDR
  record to roll up from.
- BCP: `/bcp/fincas` now shows the full checklist (not just the polygon gate)
  and blocks approval on a "no_apta" declaration, not only a missing polygon.
  New `updateFincaEudr`/`updateLotEudr` server actions + inline `<details>`
  forms on `/bcp/fincas` and `/bcp/lotes` let CTC staff fill in EUDR fields on
  a producer's behalf ("aided by BCP") the same way `bcp_manual_entry` already
  lets them register a lot for a producer. `/bcp/lotes` now shows each lot's
  live EUDR badge too.
- Deferred, not built this round: a printable/exportable DDS document (the
  reference tool's print/export button), and surfacing "EUDR Ready" as a
  buyer-facing badge on Cherry Picked's public catalog.
- Real bug caught during live verification: `fincas.requires_eudr_polygon` is
  a Postgres `GENERATED ALWAYS AS (hectares > 4) STORED` column -- an early
  version of saveFinca() tried to write to it explicitly, which made Postgres
  reject the *entire* UPDATE (400). Fixed by never sending that column;
  Postgres derives it from `hectares` on its own.

  Doc changes this should produce at the next wrap: a new node or note on
  n-kaffetal describing the EUDR dossier (finca + lote), a new DICT entry for
  the EUDR roll-up model, and a new scenario "eudr" covering producer fills
  finca+lote dossier -> BCP reviews/assists -> approval gate, through
  n-kaffetal, n-supaclient, n-guards, n-bcp, n-actions, n-db.

## Bootstrap note
This log was written when adopting the architecture-doc-versioning skill.
Earlier sessions edited the live HTML directly (the old workflow); that habit
stops here. From now on, every change is appended to this log, and the log
stays open -- nothing gets compiled into a new HTML file -- until a real
Version Wrap is invoked. A wrap takes everything in this log, applies it on
top of Documentacion_Interactiva_V1.0(b2134ed).html, and produces
Documentacion_Interactiva_V2.0(<sha>).html -- the next integer version. There
is no V1.1 or V1.2 as a file; those labels don't exist here. Once wrapped,
this file is left as-is (historical record) and a fresh
Log_Documentacion_Interactiva_V2.txt opens for whatever comes after V2.0.
