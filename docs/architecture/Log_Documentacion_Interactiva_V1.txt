## [2026-07-10] EUDR due-diligence module: Finca + Lote + BCP (commit pending)
- New `src/lib/eudr.ts`: pure, dependency-free `fincaEudrStatus()`/`lotEudrStatus()`
  ported from the reference "EUDR Ready · Café" prototype's fincaEstado()/loteEstado(),
  reinterpreted onto this system's real Finca/Lot records. Shared verbatim between
  the Kaffetal Regal client UI and BCP's server-rendered review pages.
- Finca-level (migration eudr_finca_fields): geolocation (lat/lng), fecha de
  establecimiento, sistema productivo, declaración de no deforestación + de
  producción legal, evidencia, áreas de legislación, tenencia, sostenibilidad.
  New "EUDR · Debida diligencia" section in FincaModal.tsx with a live status
  badge. Also fixed a pre-existing gap found while building this: the old
  "Geolocalización" field was never actually saved (dbFincaToFinca hardcoded
  geo:"" and saveFinca never sent it), so BCP's requires_eudr_polygon gate was
  silently inert -- now real lat/lng round-trips both ways.
- Lote-level (migration eudr_lot_fields): cadena de custodia, evaluación de
  riesgo (Art. 10(2)), mitigación -- new Ficha pane A5 ("EUDR / Debida
  Diligencia"), wired into FichaNav/FichaView alongside a live status badge.
  `additional_estates: string[]` (freetext finca names, used by Regional/
  Multi-Origin blends) renamed to `additional_estate_ids: string[]` (real
  finca ids) -- without that fix a blend's other fincas had no real EUDR
  record to roll up from.
- BCP: `/bcp/fincas` now shows the full checklist (not just the polygon gate)
  and blocks approval on a "no_apta" declaration, not only a missing polygon.
  New `updateFincaEudr`/`updateLotEudr` server actions + inline `<details>`
  forms on `/bcp/fincas` and `/bcp/lotes` let CTC staff fill in EUDR fields on
  a producer's behalf ("aided by BCP") the same way `bcp_manual_entry` already
  lets them register a lot for a producer. `/bcp/lotes` now shows each lot's
  live EUDR badge too.
- Deferred, not built this round: a printable/exportable DDS document (the
  reference tool's print/export button), and surfacing "EUDR Ready" as a
  buyer-facing badge on Cherry Picked's public catalog.
- Real bug caught during live verification: `fincas.requires_eudr_polygon` is
  a Postgres `GENERATED ALWAYS AS (hectares > 4) STORED` column -- an early
  version of saveFinca() tried to write to it explicitly, which made Postgres
  reject the *entire* UPDATE (400). Fixed by never sending that column;
  Postgres derives it from `hectares` on its own.

  Doc changes this should produce at the next wrap: a new node or note on
  n-kaffetal describing the EUDR dossier (finca + lote), a new DICT entry for
  the EUDR roll-up model, and a new scenario "eudr" covering producer fills
  finca+lote dossier -> BCP reviews/assists -> approval gate, through
  n-kaffetal, n-supaclient, n-guards, n-bcp, n-actions, n-db.

## [2026-07-11] Lot intake redesign: FT/FT2/EUDR/Video sub-stages, BCP Kanban, multi-evaluation scoring, Finca EUDR polish (commit pending)
- Root cause of a real reported bug: a lot showed FT/VID/MUE all "done" and
  already in fila_arena with zero real variety/process/score data. Cause:
  LotKanbanStepper.tsx derived its dots purely from lots.stage's numeric index,
  and nothing validated Ficha content before stage could advance past
  ficha_completa. Fixed by introducing a new `lots.intake_step` column
  (0-4) that actually gates the four producer-facing sub-stages, independent
  of `stage` -- see below.
- **Finca EUDR polish**: "Evidencia disponible", "Áreas de legislación
  verificadas", and "Sostenibilidad y enfoque social" moved to BCP-only (no
  longer producer-editable in FincaModal.tsx -- CTC staff attest these on
  /bcp/fincas as part of their own review, not the producer). "Documentos de
  respaldo" changed from a textarea to a real PDF upload
  (eudr_legal_docs_asset_id/filename, dropping the old text column). New
  FincaMapPicker.tsx: a real Google Map (pin for <=4 ha, click-to-add-vertex
  polygon for >4 ha, stored as eudr_polygon_geojson jsonb) replaces the plain
  lat/lng text inputs. Note: the polygon UI originally used
  @react-google-maps/api's DrawingManager, which crashed the browser
  (renderer-level crash, "This page couldn't load") the moment drawingMode was
  set -- replaced with a simpler manual click-to-add-vertex approach (no
  DrawingManager/drawing library dependency at all) which works correctly.
- **Lot intake sub-stages (FT/FT2/EUDR/Video)**: FichaView.tsx now has four
  sequential gates instead of one. FT (A1+A2+B1, same as the old single gate).
  FT2 (A3+A4+B2+B3, each either genuinely complete or explicitly declared "no
  lo sé / no aplica" via new ft2_*_na fields in fichaData.ts). EUDR (A5's risk
  level set, AND -- the IMPORTANT rule -- every source finca must resolve to
  "Apta" via fincaEudrStatus(), else it's blocked with an explicit message).
  Video (B4 uploaded) -- reaching this is what sets intake_step=4 and finally
  flips stage to ficha_completa, and opens a new ShipmentInstructionsModal.tsx
  (congratulations + downloadable shipment instructions, no new PDF
  dependency, just a Blob text download). FichaNav.tsx now groups panes by
  sub-stage and disables navigation into not-yet-unlocked ones; already-
  submitted sections show a "🔒 ya enviado" banner. LotKanbanStepper.tsx's
  FT/FT2/EUDR/Video dots now read `intakeStep`, not `stage` -- the actual fix
  for the reported bug. Nothing about `lot_stage`'s 7 existing values, the
  guard trigger, Arena, contracts, or catalog changed at all.
- **BCP Kanban board**: /bcp/lotes reworked from a flat list into columns (FT
  | FT2 | EUDR | Video | Muestra en tránsito), bucketed by intake_step/stage.
  No drag-and-drop library added -- progression is producer-driven, BCP's only
  action is "Confirmar recibido" (unchanged), which still moves a lot to
  fila_arena and off this board into the Arena's existing "Disponibles" list.
  Retired the old manual stage dropdown (updateLotStage) entirely -- dead code
  once intake_step drives the early stages.
- **Multi-evaluation scoring + officialization**: new `lot_evaluations` table
  (source: bcp_arena | producer_claim; status: pending/accepted/rejected). A
  lot's official Perfil de Taza / Granulometría = average of its *accepted*
  rows (src/lib/evaluations.ts, computed at read-time). BCP submits
  bcp_arena evaluations (auto-accepted) via new /bcp/evaluaciones; producer's
  own self-report is never official on its own -- a new "Solicitar
  oficialización" flow on Ficha B2 (Q-Grader reference + supporting file)
  inserts a pending producer_claim row (narrow RLS: producers may only insert
  their own pending claims for their own lots), which lands in BCP's review
  queue (accept/reject). Cherry Picked's public_lot_catalog view gained an
  official_score column (LEFT JOIN avg over accepted evaluations); the buyer
  UI now prefers it and labels the self-report "(estimado)" when no official
  score exists yet -- fixes a real pre-existing issue where the unverified
  self-report was shown indistinguishable from a verified score.

  Doc changes this should produce at the next wrap: rework the n-kaffetal
  node's Ficha description around the new 4 sub-stage model; add a new
  n-bcp-kanban concept or fold into n-bcp; add a new DICT entry for the
  multi-evaluation/officialization model; new scenario "intake" covering
  FT->FT2->EUDR->Video->shipment->BCP kanban->Arena backlog; update the
  "envio" scenario if it referenced the old single-gate model.

## [2026-07-11] Producer self-delete widened to MUE, Finca self-delete, Finca EUDR save fix (commit 1ec8dd9, 8fc14f9, c180571)
- **Real bug found and fixed**: any account without a `producer_profiles`
  row (e.g. a bcp_admin account also used to test the producer panel) had
  every "Información general" save silently no-op -- `.update().eq()`
  matches zero rows without erroring, so the UI looked saved, then the next
  full data reload pulled the still-empty row and wiped the fields back.
  Fixed with a new `producer_profiles_insert_own` RLS policy and switching
  every producer-profile write (info save, avatar/gallery/video upload) to
  `upsert` instead of `update`, so this can't silently fail for any account.
- Producer lot self-delete widened from "before FT2" to "any time before
  MUE" (before the lot passes into the Arena backlog, i.e. `stage < 4`),
  excluding `bcp_manual_entry` lots (BCP already has the physical sample in
  hand for those). New RLS policy `lots_delete_own_before_mue` replaces
  `lots_delete_own_before_ft2`.
- New: producers can delete their own Finca, but only before any EUDR
  declaration has been started (every EUDR field still blank) and only
  while no lot references it -- `lots.finca_id` is `ON DELETE CASCADE`, so
  without the second check a Finca delete could silently wipe out the
  producer's own lots too. New RLS policy `fincas_delete_own_before_eudr`;
  shared `fincaEudrUntouched()` predicate in data.ts drives both the delete
  guard and the button's visibility.
- **Real bug found and fixed**: BCP's Finca EUDR "Guardar" appeared to do
  nothing -- the save actually persisted (service-role writes bypass RLS),
  but the editor stayed in edit mode afterward with stale uncontrolled
  `defaultValue` inputs, indistinguishable from a failed save. Fixed by
  awaiting the server action explicitly and flipping back to the read view
  once it lands. `updateFincaEudr` now also diffs the submitted values
  against what was in the database and, when something actually changed,
  auto-inserts a `producer_comm_log` entry describing what changed -- every
  BCP-made EUDR edit is now logged without relying on the admin doing it
  as a separate step.

  Doc changes this should produce at the next wrap: update the producer
  self-delete note on n-guards to say "before MUE", not "before FT2"; add
  the Finca self-delete rule alongside it; mention the auto-logged BCP-edit
  trail as part of the comm-log DICT entry.

## [2026-07-12] Derived EUDR risk level, producer photo gallery, granulometry fix, dashboard reorder (commit 8e96527, plus earlier same-day commits)

- **Real bug found and fixed**: Ficha B3's granulometry (mesh) percentages
  and "Suma de mallas" total were computed against Trillado Verde Restante,
  which still includes the primary/secondary defects -- but mesh sizing is
  done on the healthy beans only, after defects are physically sorted out.
  Fixed the denominator to Grano Sano (remainder minus defects) in
  FichaView.tsx/PaneB3.tsx.
- Producer dashboard grid rebuilt to a new fixed order: Información
  general, Envío de muestras, Retroalimentación y ayuda (now spans 2 rows),

  Doc changes this should produce at the next wrap: note the derived

## Bootstrap note
This log was written when adopting the architecture-doc-versioning skill.
Earlier sessions edited the live HTML directly (the old workflow); that habit
stops here. From now on, every change is appended to this log, and the log
stays open -- nothing gets compiled into a new HTML file -- until a real
Version Wrap is invoked. A wrap takes everything in this log, applies it on
top of Documentacion_Interactiva_V1.0(b2134ed).html, and produces
Documentacion_Interactiva_V2.0(<sha>).html -- the next integer version. There
is no V1.1 or V1.2 as a file; those labels don't exist here. Once wrapped,
this file is left as-is (historical record) and a fresh
Log_Documentacion_Interactiva_V2.txt opens for whatever comes after V2.0.
