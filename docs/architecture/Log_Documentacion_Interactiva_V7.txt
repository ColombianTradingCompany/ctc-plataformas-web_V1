## [2026-07-15] Master login + parallel internal consoles (BCP / ECP / OCP)
- Reworked the internal panel to match the v3 vision board
  (reference_html-vision-board/ctc-arquitectura-v3.html). Was: one BCP sidebar with
  ECP/OCP as nested "module group" placeholder pages under /bcp/(app)/. Now: ONE
  platform master login (/login + /verify, api/panel/auth/{password,verify,logout},
  password+OTP) opening THREE parallel consoles that share one session — BCP (/bcp),
  ECP (/ecp), OCP (/ocp) — each its own route tree with a shared shell
  (src/components/panel/PanelShell+PanelSidebar, config in src/lib/panel/consoles.ts),
  a cross-console switcher, and a neutral hub at /panel. Old /bcp/login,/bcp/verify,
  /api/bcp/auth/* redirect/moved; requireAdminProfile → requireConsoleAccess(key).
- Naming correction (v3 authoritative): ECP = Executive (direction: pricing, primas,
  finances, reservations, network health); OCP = Operational (mirror of the 5 partner
  interfaces: dispatch/tracking/exceptions/relevos). The earlier scaffold had these two
  roughly swapped.
- Doc impact when wrapped:
  * New DICT concepts: "master-login" (one door, 2FA, routes by account tier),
    "consolas-paralelas" (BCP/ECP/OCP one session), "capas-de-identidad" (public
    self-signup vs internal bcp_admin vs partner tier), "matriz-de-permisos" (partner
    scoping backbone, 9 roles × 15 passport fields). Cross-link to existing BCP node.
  * New/updated NODES or node text: BCP is now "Base Control Panel" within a 3-console
    platform; ECP and OCP become first-class internal nodes (they were partner-mirror
    concepts only). Consider a small "CTC Web Platform" grouping of the 3 consoles.
  * New SCENARIOS candidate: "Login maestro → elegir consola" (login → OTP → /panel hub
    → BCP/ECP/OCP), and later "Alta de credencial de partner" once that tier is built.
  * Identity two-tier model (internal panel_users + separate partner tier) is documented
    in docs/BCP_USER_ADMIN_PLAN.md (rewritten this session) — plan, not built.
- Built + verified: tsc + eslint clean; /login renders, guarded consoles redirect to
  /login unauthenticated, old URLs forward, zero console errors. panel_users + partner
  tier remain PLAN.

- New table public.panel_users (service-role-only, RLS on, zero policies) via migration
  panel_users_internal_collaborators: profile_id PK→profiles, is_owner, consoles jsonb
  ({bcp|ecp|ocp: admin|viewer}), status invited/active/suspended, must_change_password,
  invite email tracking. Seeded the one existing founder (ctcexportmain@gmail.com) as an
  active owner with all three consoles.
- Guard: src/lib/panel/requireConsoleAccess now reads the grant (src/lib/panel/panelUsers.ts,
  service role) — status!=active ⇒ bounced to /login on next nav; console not granted ⇒ /panel;
  returns isOwner (owner-only nav group "Administración" hidden otherwise). A bcp_admin with
  no row is grandfathered (all consoles, owner) to avoid lockout.
- Master login: OTP now goes to the entering user's own email (sendOtpEmail gained a
  recipient arg; BCP_OTP_RECIPIENT_EMAIL stays as emergency override); password step rejects
  suspended collaborators; verify step stamps last_login_at and flips invited→active on first
  login.

  Resend invite email w/ temp password), suspend / reactivate (can't self-suspend or drop the
  last active owner), resend invite (regenerates temp password, invited-only). Server Actions
  in usuariosActions.ts, owner-gated + audit_log rows (entity_type 'panel_user').
- Doc impact when wrapped: strengthen the "master-login"/"capas-de-identidad" DICT entries with
  the internal-collaborator lifecycle (invite→active→suspended, per-console grants). New
  SCENARIO candidate: "Owner invita colaborador → login por-usuario → activa en primer ingreso".
